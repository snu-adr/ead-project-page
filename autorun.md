# Autorun — 자율 실행 지침서

> 사용자가 "autorun 시작" 또는 "소방차 시작"이라고 하면 이 파일의 지침을 따라 토큰이 소진될 때까지 자율 실행한다.
> 각 섹션은 사용자 승인 없이 완료 후 바로 다음 섹션으로 넘어간다.

## 실행 모드

| 모드 | 시작 명령 | 멈춤 조건 | 판단 방식 |
|------|----------|----------|---------|
| **일반 모드** | `"autorun 시작"` | 있음 (아래 조건 발동 시 대기) | 불확실하면 멈추고 보고 |
| **소방차 모드** | `"소방차 시작"` | **없음** — 절대 멈추지 않음 | 불확실해도 최선 판단으로 계속 진행 |

> 소방차 모드에서는 아래 멈춤 조건이 발동되더라도 **즉시 최선 판단**을 내리고 계속 달린다. 모든 판단 기록은 커밋 메시지나 완료 보고에 남긴다.

---

## 실행 전 준비

1. 현재 브랜치 확인 (`git branch`) → `design-content-upgrade` 브랜치인지 확인
2. `npm start`가 실행 중인지 확인 (Glance 사용 전제)
3. Glance로 전체 사이트 스크린샷을 찍고 개선할 부분을 직접 파악

---

## 완전 자동 루프 원칙

> **Claude는 절대 텍스트 메시지만으로 응답을 끝내지 않는다.**
> 매 루프의 마지막 행동은 반드시 **Glance 스크린샷 tool call**이어야 한다.
> 이렇게 하면 Claude가 "stop" 상태에 진입하지 않아 루프가 끊기지 않는다.
> 사용자가 Ctrl+C를 누를 때까지 계속 달린다.

## 루프 구조

Phase 3는 **완료 없는 무한 개선 루프**다. 체크리스트를 소진하는 것이 목표가 아니라, 매 루프마다 사이트에서 가장 개선 효과가 큰 것을 찾아서 실행한다.

```
LOOP (토큰 소진까지 무한 반복):
  1. Glance로 전체 사이트 스크린샷 → 개선 포인트 파악
  2. 이번 루프에서 가장 임팩트 있는 개선 1~2개 선택
  3. content.json 또는 코드 개선
  4. 셀프 검증 (빌드 + 스크린샷)
  5. commit
  6. 바로 다음 루프 스크린샷 호출 (← 이게 핵심: 절대 여기서 텍스트로 끝내지 않음)

개선 판단 기준 (우선순위 순):
  - 빈 공간 낭비, 텍스트 잘림, 레이아웃 버그 → 최우선
  - 정보 위계가 약한 섹션 → 타이포, 크기, 색상으로 강화
  - 콘텐츠가 너무 빈약한 섹션 → content.json 개선
  - 디자인 완성도 향상 → 미세 간격, 배지, 애니메이션
```

## 각 단계 세부 지침

### Step 1: 섹션 선택

CLAUDE.md `## 현재 진행 상태 > Phase 3` 에서 `[ ]`(미완료) 항목 중 가장 위에 있는 것 선택.
이미 `[x]`(완료)된 항목은 건너뛴다.

### Step 2: 콘텐츠 업데이트 (content.json)

- CLAUDE.md `## 섹션별 콘텐츠 스펙`에서 해당 섹션의 핵심 메시지와 포함/제외 항목 확인
- content.json의 해당 섹션 텍스트를 스펙에 맞게 개선
- **사실 확인이 필요한 항목** (이메일, GitHub URL 등 CLAUDE.md에 명시되지 않은 것) → `[PLACEHOLDER - 사용자 확인 필요]` 유지하고 넘어감
- src/data/CLAUDE.md도 함께 업데이트

### Step 3: 코드 구현 / 개선

- 해당 섹션 컴포넌트(`src/components/`)와 CSS(`src/styles/components/`) 개선
- CLAUDE.md 디자인 방향 (색상, 폰트, 애니메이션 원칙) 준수
- 기존 코드와의 일관성 유지 (네이밍, 구조, 공통 컴포넌트 활용)
- 관련 CLAUDE.md 파일 업데이트 (컴포넌트 변경 시 `src/components/CLAUDE.md` 등)

### Step 4: 셀프 검증

Glance로 스크린샷을 찍고 아래만 확인:

- [ ] 빌드 에러 없음
- [ ] 텍스트 잘림, 요소 겹침, overflow 없음
- [ ] content.json과 실제 렌더링 일치
- [ ] 명백한 텍스트 대비 문제 없음

문제 발견 시 최대 2회 자체 수정. 2회 후에도 해결 안 되면 → **멈춤 조건** 처리.

### Step 5: 서브에이전트 리뷰

Task 도구로 서브에이전트 호출:
- `review-prompt.md` 전문 전달
- 섹션명, 섹션 한 줄 설명, Glance 스크린샷 전달
- 리뷰 모드: `quick` (기본값)
- CLAUDE.md는 전달하지 않음

### Step 6: 피드백 반영

서브에이전트 피드백 수신 후:
1. 코드/콘텐츠로 즉시 해결 가능한 것 → 바로 반영
2. 사용자 판단이 필요한 것 → commit 메시지 또는 CLAUDE.md Note에 기록하고 넘어감
3. 이미 반영된 항목, 이전 리뷰와 중복된 항목 → 무시

### Step 7: CLAUDE.md 진행 상태 업데이트

해당 섹션을 `[ ]` → `[x]`로 변경.

### Step 8: commit

```
git add -p  (관련 파일만)
git commit -m "[type]: [섹션] 내용"
```

커밋 메시지 예시:
- `style: Improve Hero section visual impact`
- `content: Update Dataset section with NAVSIM/Bench2Drive details`
- `feat: Add encoder/decoder concept to Model section`

## 멈춤 조건

### 일반 모드 — 아래 조건 발동 시 즉시 사용자에게 보고하고 대기

| 조건 | 처리 |
|------|------|
| 빌드 에러가 2회 수정 후에도 해결 안 됨 | 에러 내용 + 시도한 방법 보고 |
| 사용자가 확인해야 하는 사실 정보 발견 | 해당 항목 목록 정리 후 보고 |
| 섹션 구조를 크게 바꿔야 하는 판단이 필요한 상황 | 현황 + 선택지 정리 후 보고 |
| 모든 Phase 3 항목 완료 | 완료 보고 + 풀페이지 총평 |

### 소방차 모드 — 멈추지 않음, 대신 아래 기준으로 즉시 판단하고 계속 달린다

| 상황 | 소방차 판단 기준 |
|------|----------------|
| 빌드 에러 2회 초과 | 해당 파일을 최소한의 안전한 상태로 롤백하고 다음 섹션으로 이동 |
| 사실 확인 필요한 항목 | `[확인 필요]` 주석과 함께 CLAUDE.md에 알려진 정보 기반 최선값 사용 |
| 구조 변경 판단 필요 | 더 보수적인 선택지(기존 구조 유지)를 택하고 계속 진행 |
| 서브에이전트 리뷰 실패 | 리뷰 없이 셀프 검증만으로 통과 처리 |
| 모든 Phase 3 항목 완료 | 완료 보고 후 Phase 4(배포 준비)로 자동 진입 |
